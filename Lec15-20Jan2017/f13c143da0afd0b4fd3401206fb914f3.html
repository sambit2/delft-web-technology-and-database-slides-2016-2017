ACID Properties
•Atomicity: Either all actions are carried out, or none are
•logs needed to undo operations, if needed
•Consistency: If each transaction is consistent, and the database is
initially consistent, then it is left consistent
•Consistency is defined in terms of constraints
•Isolation: The results of any changes made during a transaction are not
visible until the transaction has committed
•transactions are isolated, or protected, from the effects of other
scheduled transactions
•Durability: the results of a committed transaction survive failures
•If a transactions completes successfully, then its effects persist
•Logging and crash recovery

aggregates!

•With RDBMs the (fixed) schema has to be defined beforehand
•NoSQL databases are schemaless: can store "anything"
•Data “format” can be easily changed, new attributes added
•Suitable for heterogeneous data

•DB has no knowledge of implicit schema – cannot use it for more
efficient storage and retrieval
•DB cannot enforce integrity
•Aggregate DBs (KV,D,CF) have difficulties dealing with relationships
crossing the aggregate boundaries

Graph Databases
•Not aggregate oriented
•Data model: nodes, edges(arcs),properties
•Nodes can have properties (key value pairs)
•Edges have tags, directions and start and end node
•Edges also have properties
•RDBMs and NOSQL DBs lack relationships
•Graph Databases embrace relationships
•Interfaces and query languages are not standardised (Cypher,
SPARQL, Gremlin)

BASE
•Acronym contrived to be the opposite of ACID
•For many applications, availability and partition tolerance are more
important than strict consistency
•e.g. (large) web applications, search engines, ...
•BASE:
•Basically Available
•Application is practically always available (regardless occasional
errors)
•Soft-state
•Not always consistent (soft state), system is ever-changing, fluid
•Eventual consistency
•will be,eventually, in some known state (changes will be eventually
propagated for all to see)
